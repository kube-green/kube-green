Introducción

La API REST de Kube-Green proporciona una interfaz programática para gestionar las configuraciones de apagado y encendido automático de recursos Kubernetes. Esta documentación está diseñada para desarrolladores e ingenieros de DevOps que necesitan integrar la gestión de schedules en sus pipelines de CI/CD.

URL Base

La API está disponible en la siguiente URL base:

# Desde dentro del cluster Kubernetes BASE_URL = "http://kube-green-api-service.keos-core.svc.cluster.local:8080" # Desde fuera del cluster (con port-forward) BASE_URL = "http://localhost:8080"
ℹ️
Nota Importante
Para acceder desde fuera del cluster, utiliza kubectl port-forward para exponer el servicio localmente.

Formato de Respuestas

Todas las respuestas de la API siguen un formato estándar JSON:

Respuesta Exitosa

{ "success": true, "message": "Operación completada exitosamente", "data": { // Datos de la respuesta } }

Respuesta de Error

{ "success": false, "error": "Mensaje de error descriptivo", "code": 400 }
⚠️
Importante
Siempre verifica el campo success antes de procesar los datos en tus scripts.

Autenticación

La API utiliza autenticación basada en tokens JWT (JSON Web Tokens). Todos los endpoints, excepto los de autenticación y documentación, requieren un token Bearer válido en el header Authorization.

Flujo de Autenticación

1
Realizar Login

Envía una petición POST al endpoint de login con tus credenciales.

POST /api/v1/auth/login
# Ejemplo con curl curl -X POST "${BASE_URL}/api/v1/auth/login" \ -H "Content-Type: application/json" \ -d '{ "username": "admin", "password": "admin123" }'
2
Obtener Token de Acceso

La respuesta incluye un token de acceso y un token de refresco.

{ "success": true, "data": { "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "expiresIn": 86400 } }
3
Usar el Token en Peticiones

Incluye el token en el header Authorization de todas las peticiones protegidas.

# Formato del header Authorization: Bearer ${TOKEN} # Ejemplo completo curl -X GET "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json"

Refrescar Token

Cuando el token esté próximo a expirar, puedes obtener uno nuevo usando el refresh token:

POST /api/v1/auth/refresh
Renueva el token de acceso usando el refresh token almacenado.
curl -X POST "${BASE_URL}/api/v1/auth/refresh" \ -H "Content-Type: application/json" \ -d '{ "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." }'

Roles y Permisos

La API implementa un sistema de roles con diferentes niveles de acceso:

Rol Descripción Permisos
admin Administrador Acceso completo a todos los endpoints, incluyendo gestión de usuarios
operacion Operación Gestión completa de schedules (crear, editar, eliminar, ver)
lectura Lectura Solo lectura de schedules y tenants

Endpoints de la API

Gestión de Tenants

GET /api/v1/tenants
Lista todos los tenants descubiertos en el cluster Kubernetes.
Autenticación

Requerida - Todos los roles

curl -X GET "${BASE_URL}/api/v1/tenants" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json"

Gestión de Schedules

Listar Todos los Schedules

GET /api/v1/schedules
Obtiene todos los schedules configurados en todos los tenants.
curl -X GET "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}"

Obtener Schedule de un Tenant

GET /api/v1/schedules/{tenant}
Obtiene la configuración de schedule para un tenant específico.
Parámetros
Parámetro Tipo Ubicación Requerido Descripción
tenant string Path Sí Nombre del tenant (ej: "bdadevdat")
namespace string Query No Filtro por namespace específico
# Obtener schedule de un tenant específico TENANT="bdadevdat" curl -X GET "${BASE_URL}/api/v1/schedules/${TENANT}" \ -H "Authorization: Bearer ${TOKEN}" # Obtener schedule de un namespace específico curl -X GET "${BASE_URL}/api/v1/schedules/${TENANT}?namespace=datastores" \ -H "Authorization: Bearer ${TOKEN}"

Crear Schedule

POST /api/v1/schedules
Crea un nuevo schedule para un tenant. Requiere permisos de administrador o operación.
Body Parameters
Campo Tipo Requerido Descripción
tenant string Sí Nombre del tenant
scheduleName string Sí Nombre identificador del schedule
description string No Descripción del schedule
off string Sí Hora de apagado (formato HH:MM, 24 horas)
on string Sí Hora de encendido (formato HH:MM, 24 horas)
sleepDays string Sí Días de apagado (ej: "lunes,martes,miércoles,jueves,viernes")
wakeDays string Sí Días de encendido
namespaces array Sí Lista de namespaces a gestionar
userTimezone string Sí Zona horaria del usuario (ej: "America/Bogota")
clusterTimezone string Sí Zona horaria del cluster (generalmente "UTC")
curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "horario-laboral", "description": "Apagado nocturno de lunes a viernes", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "22:00", "on": "06:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes", "wakeDays": "lunes,martes,miércoles,jueves,viernes", "namespaces": ["datastores", "apps", "rocket"], "delays": { "pgbouncerDelay": "0m", "deploymentsDelay": "5m" }, "exclusions": [ { "namespace": "apps", "resourceType": "Deployment", "resourceName": "virtualizer" } ] }'
ℹ️
Notas Importantes
  • Los campos off y on deben estar en formato 24 horas (HH:MM)
  • sleepDays y wakeDays pueden ser días en español o números (0-6, donde 0=domingo)
  • La API convierte automáticamente las horas de la zona del usuario a UTC

Casos de Uso Prácticos

Esta sección contiene ejemplos prácticos de cómo configurar schedules para diferentes escenarios comunes en entornos de producción y desarrollo.

1. Apagado de un Tenant Completo

Configura un schedule para apagar todos los namespaces de un tenant específico:

# Ejemplo: Apagar tenant "bdadevdat" todos los días a las 22:00 curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "apagado-completo-tenant", "description": "Apagado completo del tenant todas las noches", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "22:00", "on": "06:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes,sábado,domingo", "wakeDays": "lunes,martes,miércoles,jueves,viernes,sábado,domingo", "namespaces": ["datastores", "apps", "rocket", "intelligence", "airflowsso"] }'

2. Apagado Solo Días Laborales (Lunes a Viernes)

Apaga los servicios solo de lunes a viernes, manteniéndolos encendidos los fines de semana:

# Ejemplo: Apagar solo días laborales curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "horario-laboral", "description": "Apagado nocturno solo días laborales", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "22:00", "on": "06:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes", "wakeDays": "lunes,martes,miércoles,jueves,viernes", "namespaces": ["datastores", "apps", "rocket"] }'

3. Apagado de Fin de Semana

Apaga los servicios solo los fines de semana (sábado y domingo):

# Ejemplo: Apagar solo fines de semana curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "apagado-fin-semana", "description": "Apagado completo durante fines de semana", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "20:00", "on": "08:00", "sleepDays": "sábado,domingo", "wakeDays": "sábado,domingo", "namespaces": ["datastores", "apps", "rocket"] }'
ℹ️
Nota sobre Fines de Semana
El apagado comienza el viernes a las 20:00 y el encendido el lunes a las 08:00. Los días de sleep y wake deben incluir sábado y domingo.

4. Apagado de un Namespace Específico

Configura un schedule para apagar solo un namespace específico dentro de un tenant:

# Ejemplo: Apagar solo el namespace "apps" curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "apagado-namespace-apps", "description": "Apagado solo del namespace apps", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "23:00", "on": "05:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes", "wakeDays": "lunes,martes,miércoles,jueves,viernes", "namespaces": ["apps"] }'

5. Apagado de Varios Namespaces con Excepciones

Apaga varios namespaces pero excluye recursos específicos:

# Ejemplo: Apagar múltiples namespaces excluyendo recursos específicos curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "apagado-selectivo", "description": "Apagado de múltiples namespaces con exclusiones", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "22:00", "on": "06:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes", "wakeDays": "lunes,martes,miércoles,jueves,viernes", "namespaces": ["datastores", "apps", "rocket"], "exclusions": [ { "namespace": "apps", "resourceType": "Deployment", "resourceName": "virtualizer" }, { "namespace": "datastores", "resourceType": "StatefulSet", "resourceName": "postgres-primary" } ] }'

6. Apagado con Delays Escalonados

Configura delays para que los servicios se apaguen y enciendan en orden específico:

# Ejemplo: Apagado con delays para ordenar el encendido curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "apagado-con-delays", "description": "Apagado con delays escalonados para encendido ordenado", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "22:00", "on": "06:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes", "wakeDays": "lunes,martes,miércoles,jueves,viernes", "namespaces": ["datastores", "apps"], "delays": { "pgbouncerDelay": "0m", "deploymentsDelay": "5m" } }'
ℹ️
Sobre los Delays
  • pgbouncerDelay: Tiempo de espera antes de encender PgBouncer (ej: "0m", "2m", "5m")
  • deploymentsDelay: Tiempo de espera antes de encender los Deployments (ej: "5m", "10m")
  • Los delays aseguran que los servicios se enciendan en el orden correcto (primero bases de datos, luego aplicaciones)

7. Apagado Todos los Días (24/7 con Horario Específico)

Apaga los servicios todos los días a una hora específica:

# Ejemplo: Apagado todos los días a las 23:00 curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "apagado-diario", "description": "Apagado todos los días a las 23:00", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "23:00", "on": "07:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes,sábado,domingo", "wakeDays": "lunes,martes,miércoles,jueves,viernes,sábado,domingo", "namespaces": ["datastores", "apps", "rocket"] }'

8. Múltiples Schedules para el Mismo Tenant

Puedes crear múltiples schedules con diferentes configuraciones usando diferentes scheduleName:

# Schedule 1: Apagado días laborales curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "horario-laboral", "description": "Apagado días laborales", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "22:00", "on": "06:00", "sleepDays": "lunes,martes,miércoles,jueves,viernes", "wakeDays": "lunes,martes,miércoles,jueves,viernes", "namespaces": ["apps", "rocket"] }' # Schedule 2: Apagado fin de semana (mismo tenant, diferente scheduleName) curl -X POST "${BASE_URL}/api/v1/schedules" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "tenant": "bdadevdat", "scheduleName": "horario-fin-semana", "description": "Apagado fin de semana", "userTimezone": "America/Bogota", "clusterTimezone": "UTC", "off": "20:00", "on": "08:00", "sleepDays": "sábado,domingo", "wakeDays": "sábado,domingo", "namespaces": ["datastores", "apps"] }'

9. Actualizar un Schedule Existente

Para modificar un schedule existente, usa PUT con solo los campos que quieres cambiar:

# Ejemplo: Cambiar solo la hora de apagado curl -X PUT "${BASE_URL}/api/v1/schedules/bdadevdat" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "off": "23:30", "on": "06:30" }' # Ejemplo: Agregar un namespace adicional curl -X PUT "${BASE_URL}/api/v1/schedules/bdadevdat" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json" \ -d '{ "namespaces": ["datastores", "apps", "rocket", "intelligence"] }'

10. Consultar Schedule Configurado

Para ver la configuración actual de un tenant:

# Obtener todos los schedules de un tenant curl -X GET "${BASE_URL}/api/v1/schedules/bdadevdat" \ -H "Authorization: Bearer ${TOKEN}" # Obtener schedule de un namespace específico curl -X GET "${BASE_URL}/api/v1/schedules/bdadevdat?namespace=apps" \ -H "Authorization: Bearer ${TOKEN}"

11. Eliminar un Schedule

Para eliminar todos los schedules de un tenant:

# Eliminar todos los schedules del tenant curl -X DELETE "${BASE_URL}/api/v1/schedules/bdadevdat" \ -H "Authorization: Bearer ${TOKEN}"
⚠️
Advertencia
Eliminar un schedule detendrá todos los apagados/encendidos programados para ese tenant. Los servicios actualmente apagados permanecerán apagados hasta que se enciendan manualmente.

12. Formato de Días de la Semana

Puedes usar días en español o números (0=domingo, 1=lunes, ..., 6=sábado):

Formato Ejemplo Descripción
Español "lunes,martes,miércoles" Días escritos en español (recomendado)
Números "1,2,3" 0=domingo, 1=lunes, 2=martes, ..., 6=sábado
Rango "1-5" Equivale a lunes a viernes
Mixto "lunes,3,viernes" Puedes mezclar formatos

Ejemplos para CI/CD

Script Bash Completo

Ejemplo completo de script bash para gestionar schedules desde pipelines:

#!/bin/bash # Script completo para gestionar schedules desde CI/CD # ============================================================================ # Configuración # ============================================================================ BASE_URL="${KUBE_GREEN_API_URL:-http://localhost:8080}" USERNAME="${KUBE_GREEN_USERNAME:-admin}" PASSWORD="${KUBE_GREEN_PASSWORD:-admin123}" # ============================================================================ # Funciones # ============================================================================ # Función para obtener token get_token() { local response=$(curl -s -X POST "${BASE_URL}/api/v1/auth/login" \ -H "Content-Type: application/json" \ -d "{\"username\":\"${USERNAME}\",\"password\":\"${PASSWORD}\"}") local success=$(echo "$response" | jq -r '.success') if [ "$success" != "true" ]; then echo "Error: Fallo en autenticación" >&2 echo "$response" | jq '.' >&2 exit 1 fi echo "$response" | jq -r '.data.accessToken' } # Función para hacer peticiones autenticadas api_request() { local method=$1 local endpoint=$2 local data=$3 local url="${BASE_URL}${endpoint}" local cmd=(curl -s -X "$method" "$url" \ -H "Authorization: Bearer ${TOKEN}" \ -H "Content-Type: application/json") if [ -n "$data" ]; then cmd+=("-d" "$data") fi "${cmd[@]}" } # ============================================================================ # Ejecución # ============================================================================ # Obtener token echo "Obteniendo token de autenticación..." TOKEN="$(get_token)" if [ -z "$TOKEN" ] || [ "$TOKEN" = "null" ]; then echo "Error: No se pudo obtener el token" >&2 exit 1 fi echo "✓ Token obtenido exitosamente" # Ejemplo: Listar tenants echo "Listando tenants..." response="$(api_request GET "/api/v1/tenants")" echo "$response" | jq '.'

Script Python

Clase Python reutilizable para interactuar con la API:

#!/usr/bin/env python3 """ Clase para interactuar con la API de Kube-Green """ import os import json import requests from typing import Optional, Dict, Any class KubeGreenAPI: """Cliente para la API REST de Kube-Green""" def __init__(self, base_url: str, username: str, password: str): self.base_url = base_url.rstrip('/') self.username = username self.password = password self.token = None self.refresh_token = None def login(self) -> bool: """Autentica y obtiene el token de acceso""" url = f"{self.base_url}/api/v1/auth/login" data = { "username": self.username, "password": self.password } try: response = requests.post(url, json=data, timeout=10) response.raise_for_status() result = response.json() if result.get('success'): self.token = result['data']['accessToken'] self.refresh_token = result['data']['refreshToken'] return True else: print(f"Error en login: {result.get('error')}") return False except requests.exceptions.RequestException as e: print(f"Error de conexión: {e}") return False def _request(self, method: str, endpoint: str, data: Optional[Dict] = None) -> Dict[Any, Any]: """Realiza una petición autenticada""" if not self.token: raise ValueError("No hay token. Llama a login() primero") url = f"{self.base_url}{endpoint}" headers = { "Authorization": f"Bearer {self.token}", "Content-Type": "application/json" } try: if method.upper() == 'GET': response = requests.get(url, headers=headers, timeout=30) elif method.upper() == 'POST': response = requests.post(url, headers=headers, json=data, timeout=30) elif method.upper() == 'PUT': response = requests.put(url, headers=headers, json=data, timeout=30) elif method.upper() == 'DELETE': response = requests.delete(url, headers=headers, timeout=30) else: raise ValueError(f"Método no soportado: {method}") response.raise_for_status() return response.json() except requests.exceptions.RequestException as e: print(f"Error en petición: {e}") if hasattr(e, 'response') and e.response is not None: print(f"Respuesta: {e.response.text}") raise def list_tenants(self) -> Dict[Any, Any]: """Lista todos los tenants""" return self._request('GET', '/api/v1/tenants') def create_schedule(self, tenant: str, schedule_config: Dict) -> Dict[Any, Any]: """Crea un nuevo schedule""" schedule_config['tenant'] = tenant return self._request('POST', '/api/v1/schedules', schedule_config) # Ejemplo de uso if __name__ == "__main__": api = KubeGreenAPI( base_url=os.getenv('KUBE_GREEN_API_URL', 'http://localhost:8080'), username=os.getenv('KUBE_GREEN_USERNAME', 'admin'), password=os.getenv('KUBE_GREEN_PASSWORD', 'admin123') ) if not api.login(): print("Error: No se pudo autenticar") exit(1) print("✓ Autenticación exitosa") # Listar tenants tenants = api.list_tenants() print(f"Tenants encontrados: {len(tenants['data']['tenants'])}")

Azure DevOps Pipeline

Pipeline completo para Azure DevOps con manejo de errores:

# azure-pipelines.yml trigger: - main pool: vmImage: 'ubuntu-latest' variables: - name: kubeGreenApiUrl value: 'http://kube-green-api-service.keos-core.svc.cluster.local:8080' - name: tenant value: 'bdadevdat' stages: - stage: DeploySchedule displayName: 'Desplegar Schedule' jobs: - job: Deploy displayName: 'Desplegar Schedule de Kube-Green' steps: - task: Bash@3 displayName: 'Autenticar y obtener token' inputs: targetType: 'inline' script: | LOGIN_RESPONSE=$(curl -s -X POST "$(kubeGreenApiUrl)/api/v1/auth/login" \ -H "Content-Type: application/json" \ -d "{\"username\":\"$(KUBE_GREEN_USERNAME)\",\"password\":\"$(KUBE_GREEN_PASSWORD)\"}") SUCCESS=$(echo "$LOGIN_RESPONSE" | jq -r '.success') if [ "$SUCCESS" != "true" ]; then echo "Error en autenticación" exit 1 fi TOKEN=$(echo "$LOGIN_RESPONSE" | jq -r '.data.accessToken') echo "##vso[task.setvariable variable=KUBE_GREEN_TOKEN]$TOKEN" env: KUBE_GREEN_USERNAME: $(KUBE_GREEN_USERNAME) KUBE_GREEN_PASSWORD: $(KUBE_GREEN_PASSWORD)
ℹ️
Configuración de Secretos
Configura las variables KUBE_GREEN_USERNAME y KUBE_GREEN_PASSWORD como secretos en las variables del pipeline.

Manejo de Errores

Códigos de Estado HTTP

Código Significado Acción Recomendada
200 Éxito Procesar respuesta normalmente
400 Bad Request Verificar formato de la petición y parámetros
401 No autorizado Token inválido o expirado. Renovar token
403 Prohibido Usuario no tiene permisos suficientes
404 No encontrado Recurso no existe
500 Error interno Error del servidor. Reintentar o contactar soporte

Mejores Prácticas

Seguridad

  • Nunca hardcodees credenciales: Usa variables de entorno o secretos de tu sistema CI/CD
  • Valida siempre las respuestas: Nunca asumas que una petición fue exitosa sin verificar
  • Maneja la expiración de tokens: Implementa renovación automática de tokens

Robustez

  • Configura timeouts en tus peticiones HTTP
  • Implementa lógica de reintento para errores 500 o de red
  • Registra todas las operaciones importantes para debugging
  • Valida los datos antes de enviarlos a la API
